● Kritische Performance-Probleme

6 Issues mit direktem Impact auf User Experience

Diese Probleme verursachen spuerbare Verzoegerungen bei Interaktionen und skalieren schlecht mit steigender Nutzung.

K1

JS-Interop bei jedem Mausbewegungsevent

WhiteboardCanvas.Interaction.cs:18-23

GetScreenPointerAsync() fuehrt bei jedem mousemove-Event einen JS-Interop-Aufruf ueber SignalR durch. Bei 60fps = 60 Roundtrips/Sekunde.

⚠ 200-500ms Latenz bei Drag
Kritisch
Vorher
// Jedes mousemove = SignalR-Roundtrip
private async Task<Point> GetScreenPointerAsync(MouseEventArgs e)
{
    var pointer = await JS.InvokeAsync<RelativePointer>(
        "orimWhiteboard.clientToElement",
        SurfaceId, e.ClientX, e.ClientY);
}
Nachher
// Offset einmal cachen, kein JS-Interop
private Point GetScreenPointer(MouseEventArgs e)
{
    return ClampToSurface(new Point(
        e.ClientX - _surfaceOffset.X,
        e.ClientY - _surfaceOffset.Y));
}
K2

LINQ OrderBy im Render-Template bei jedem Frame

WhiteboardCanvas.razor:21

Board.Elements.OrderBy(e => e.ZIndex) wird bei jedem Render ausgefuehrt — auch waehrend Drag-Operationen mit Dutzenden Renders/Sekunde.

⚠ 2-8ms pro Render
Hoch
Vorher
// O(n log n) Sortierung bei JEDEM Render
@foreach (var element in
    Board.Elements.OrderBy(e => e.ZIndex))
{
    @RenderElement(element)
}
Nachher
// Sortierte Liste cachen
@foreach (var element in _sortedElements)
{
    <div @key="element.Id">
        @RenderElement(element)
    </div>
}

● Rendering-Ineffizienzen

Unnoetige Re-Renders und DOM-Operationen

Blazors Diff-Algorithmus wird nicht optimal genutzt. Jede Interaktion triggert mehr DOM-Updates als noetig.

R1

Fehlende @key Directive auf Board-Elementen

WhiteboardCanvas.razor:21-24

Ohne @key muss Blazor sequenziell diffieren. Wird ein Element eingefuegt, werden alle folgenden als "geaendert" betrachtet.

▲ +15% Rendering mit @key
R2

Inline-Styles bei jedem Render neu allokiert

WhiteboardCanvas.Rendering.cs:11-18

GetSurfaceStyle() baut einen ~400 Zeichen String bei jedem Render zusammen. Sollte gecacht und nur bei Zoom/Pan/Theme-Aenderung aktualisiert werden.

R3

RenderFragment-Delegates pro Element pro Render

WhiteboardCanvas.Rendering.cs:158-203

Jeder RenderElement-Aufruf erzeugt neue Lambda-Delegates. Bei 50 Elementen = 50 Allokationen pro Render.

R4

Mehrfache StateHasChanged() in Sequenz

BoardEditor.razor:622-633

HandleChatBoardChangedAsync ruft 3x StateHasChanged() in einer Methode auf — ueber ForceRerender(), FitToContentAsync() und direkt.

R5

FilteredMaterialIcons: LINQ bei jedem Zugriff

BoardEditor.razor:529-534

Computed Property evaluiert 7.000+ Icons mit Where, OrderBy, ThenBy, Take, ToList bei jedem Render.

R6

Board-Referenz als Parameter triggert Re-Render

BoardEditor.razor:286-295

Board als [Parameter] an WhiteboardCanvas: Blazor ruft SetParametersAsync bei jedem Parent-Render auf. Kein ShouldRender() implementiert.

● Optimierungs-Opportunitaeten

Quick Wins und strategische Verbesserungen

O1

Client-seitiges Throttle fuer Maus-Events

JavaScript-seitig mousemove auf 60fps (16ms) begrenzen. Nur letzte Position an Server senden. Reduziert SignalR-Traffic um 80-90%.

O2

Singleton-SemaphoreSlim blockiert alle Operationen

JsonBoardRepository.cs:12

Ein einzelnes Lock serialisiert alle Board-Operationen. User A speichert → User B wartet. Per-Board-Locking wuerde Concurrency ermoeglichen.

O3

AI-Prompt mit vollem Board-JSON

DiagramAssistantService.cs:136-183

Das gesamte Board wird als JSON in den System-Prompt serialisiert. Bei grossen Boards = tausende unnoetige Tokens pro AI-Request.

O4

Keine ErrorBoundary um kritische Komponenten

Ein Fehler in WhiteboardCanvas bringt die gesamte Seite zum Absturz. <ErrorBoundary> wuerde Fehler isolieren und Reconnection vermeiden.

O5

Kein Virtual Scrolling im Icon-Picker

BoardEditor.razor:365-374

240 Icon-Buttons gleichzeitig gerendert. Virtualize<T> wuerde nur sichtbare Items rendern.

O6

Lambda-Closures in foreach-Schleifen

Home.razor:39, BoardEditor.razor:55-56

Inline-Lambdas in @foreach erzeugen neue Delegates bei jedem Render. Child-Komponenten mit @key waeren effizienter.

O7

CommandStack ohne History-Limit

CommandStack.cs:80-116

Unbegrenzter Undo-Stack. Bei intensiver Nutzung waechst der Speicherverbrauch im Circuit kontinuierlich. Limit auf 50 Schritte empfohlen.

● Erkannte AI-Code-Patterns

Typische Muster von AI-generiertem Code

Diese Anti-Patterns treten haeufig bei AI-generiertem Blazor-Code auf und sind in diesem Projekt identifizierbar.

Identifizierte AI-Code-Muster in ORIM
  • 25+ private State-Variablen in BoardEditor statt strukturiertem ViewModel
  • Direkte Event-Handler in Parent-Komponenten statt Child-Abstraktion
  • Keine Performance-Optimierungen (kein @key, kein ShouldRender, kein Caching)
  • Vollstaendige Objekte als Parameter statt gezielter IDs/Selektoren
  • Fehlende Pagination und Virtual Scrolling bei Listen
  • LINQ-Ketten in Render-Pfaden statt vorberechneter Ergebnisse
